#+TITLE: Dagger caches by operation graph, not content
#+LANGUAGE: en
#+PROPERTY: header-args:bash :tangle no :results verbatim :cache yes

Dagger's directory layer is content-addressed: two directories built
via completely different operation chains produce the same =digest()=
if their files are byte-for-byte identical.  However, exec caching
is graph-based: when the directory is mounted into a container, the
=withExec= cache key includes the full operation lineage — not the
content.  Two content-identical directories reached via different
graph paths produce different container IDs, and every =withExec=
downstream is a cache miss.

This study builds a minimal reproducer, proves the behavior, and
links to the upstream fix.

* The module

  A standalone Dagger module with no dependencies:

  #+BEGIN_SRC bash :results none :exports code
    dagger init --sdk=python
  #+END_SRC


  #+begin_src python :tangle .dagger/src/caching/__init__.py :eval no-export

  The module defines two helper functions that build the same
  directory — a single file =data.txt= containing =hello= — via
  different operation graphs:

  - =_dir_a()= creates the directory directly.
  - =_dir_b()= creates the same directory, adds a =junk.txt=, then

* Proving the directories are identical but containers are not

  =prove_identical= computes the content digest of both directories
  and the container ID after mounting each one.  It also tests three
  attempts to "flatten" the graph:

  - =sync()= — resolve the directory eagerly
  - copy through =with_directory(".", ...)= — rewrap in a fresh directory
  - round-trip through a container — mount, then extract with =.directory()=

  #+NAME: prove-identical
  #+begin_src python
  @function
  async def prove_identical(self) -> str:
      """Show that dir_a and dir_b have the same content digest but different container IDs."""
      dir_a = _dir_a()
      dir_b = _dir_b()

      digest_a = await dir_a.digest()
      digest_b = await dir_b.digest()

      ctr = dag.container().from_("alpine:latest")

      ctr_a = ctr.with_directory("/work", dir_a)
      ctr_b = ctr.with_directory("/work", dir_b)
      ctr_c = ctr.with_directory("/work", dir_a)

      # Attempt to flatten the graph
      dir_b_synced = await dir_b.sync()
      ctr_b_synced = ctr.with_directory("/work", dir_b_synced)

      dir_b_copied = dag.directory().with_directory(".", dir_b)
      ctr_b_copied = ctr.with_directory("/work", dir_b_copied)

      dir_b_roundtrip = (
          dag.container()
          .from_("alpine:latest")
          .with_directory("/work", dir_b)
          .directory("/work")
      )
      ctr_b_roundtrip = ctr.with_directory("/work", dir_b_roundtrip)

      id_a = await ctr_a.id()
      id_b = await ctr_b.id()
      id_c = await ctr_c.id()
      id_b_synced = await ctr_b_synced.id()
      id_b_copied = await ctr_b_copied.id()
      id_b_roundtrip = await ctr_b_roundtrip.id()

      return (
          f"dir digests identical: {digest_a == digest_b}\n"
          f"\n"
          f"a vs b (different graph, same content): {id_a == id_b}\n"
          f"a vs c (same graph):                    {id_a == id_c}\n"
          f"\n"
          f"flattening attempts (all should be True to fix caching):\n"
          f"  sync():                {id_a == id_b_synced}\n"
          f"  copy via with_directory: {id_a == id_b_copied}\n"
          f"  roundtrip via container: {id_a == id_b_roundtrip}"
      )
  #+end_src

* Observing the exec cache bust

  =cached= and =busted= mount =dir_a= and =dir_b= respectively into
  an alpine container and run the same slow command.

  Each takes a ~run~ argument so that re-calling with a different
  value forces the function to re-execute; that lets us observe the
  underlying =withExec= cache, not just the module-level result
  cache.

  #+NAME: cached
  #+begin_src python
  @function
  async def cached(self, run: int = 0) -> str:
      """Mount dir_a and run a slow command. Returns withExec elapsed seconds."""

          .from_("alpine:latest")
          .with_directory("/work", _dir_a())
          .with_exec(["sh", "-c", "sleep 3 && echo done"])

* Comparing workarounds

  The sections above prove the problem with synthetic directories.
  Now we test whether =DefaultPath(".")= + =Ignore= avoids the
  cache bust when applied to real context directories with real
  files changing.

  The behavior depends on the module layout, so we test two
  sub-projects side by side:

  - =with-dagger-source/= — uses =source: .dagger= (module source
    is =.dagger/=, isolated from the project root)
  - =with-root-source/= — no ="source"= field (module source is
    the project root, like daggerlib's =lib= module)

  Each sub-project has the same two functions:

  - =old_way()= uses =dag.current_module().source()= +
    =with_directory(include=["data.txt"])= — filtering inside the
    function.
  - =new_way(src)= uses =DefaultPath(".")= +
    =Ignore(["**", "!data.txt"])= — filtering at the engine level.

** Test protocol

   Each sub-project is tested: populate cache, confirm hit, add a
   junk file where it changes the module source, re-run.  =time=
   makes the result visible: ~1 s = cached, ~20 s = busted.

   The cache is pruned before each test group with =dagger core
   engine local-cache prune= (without =--use-default-policy=, which
   removes all releasable entries).  Three runs per group: populate,
   confirm cached, add junk and re-run.

   Wall-clock =time= is misleading: every call pays ~10 s of
   unavoidable module reload when a source file changes.  The
   crucial signal is the exec duration inside the function.  Each
   Dagger function measures its own =withExec= duration with
   =time.monotonic()= and returns it as a string:

   - =~0.03s= — exec cache survived (no =sleep= ran)
   - =~3.10s= — exec re-ran (=sleep 3=)

   Each block prunes the cache, populates it with a fresh call, adds
   a junk file inside the module source, re-runs, and prints the
   measured exec duration.

*** Conclusion

    Both layouts show the same result: =old_way= busts (~3.5 s —
    =sleep 3= runs again), =new_way= stays cached (~0.2 s — exec
    cache survives).

    | Layout | Approach | withExec duration | Cache  |
    |--------+----------+-------------------+--------|
    | =.dagger= | old_way  | ~3.6 s            | BUSTED |
    | =.dagger= | new_way  | ~0.2 s            | CACHED |
    | root   | old_way  | ~3.5 s            | BUSTED |
    | root   | new_way  | ~0.2 s            | CACHED |

    The =DefaultPath(".")= + =Ignore(["**", "!..."])= workaround
    reliably preserves the exec cache regardless of module layout,
    as long as junk files don't match the allowlist patterns.

* Root cause

  Dagger has two caching layers:

  - *dagql* (Dagger's own layer) is content-addressed.
    =Directory.digest()= returns a content hash, and two directories
    with identical content produce the same digest.

  - *BuildKit's solver* (the exec layer) is graph-based.  The cache
    key for =withExec= includes the full operation chain that built
    the container, not the content of its filesystem.  Two containers
    with identical filesystems but different build histories get
    different cache keys.

  The mismatch means that =withDirectory= correctly deduplicates
  content, but the container identity still encodes the graph path.
  Any =withExec= downstream sees a "new" container and re-runs.

  A Dagger contributor traced the specific mechanism in
  [[https://github.com/dagger/dagger/issues/8955][issue #8955]]: when local source directories are unfrozen (since
  [[https://github.com/dagger/dagger/pull/8818][PR #8818]]), the LLB operations use a =session:= prefix that changes
  between runs even when content is identical.  The dagql IDs are
  correct (content-hashed via =MakeDirectoryContentHashed=) but the
  BuildKit LLB is not.

** Why BuildKit does not content-dedup exec cache keys

   BuildKit's solver has two cache key modes for exec operations:
   a fast mode (graph-based, using operation identity) and a slow
   mode (content-based, using filesystem checksums).  The slow
   content-based mode is [[https://github.com/moby/buildkit/blob/a09112603a36532b8e44ade74ee3fd8383f4b708/solver/llbsolver/ops/exec.go#L260-L262][explicitly disabled in =exec.go=]] for
   mounts that are either the root mount or non-readonly.  The
   rationale ([[https://github.com/moby/buildkit/issues/903#issuecomment-475923526][moby/buildkit#903]]): computing content-based
   checksums for every mount is expensive, so BuildKit only pays
   that cost for non-root, read-only mounts.

   This is fundamental to BuildKit, not Dagger-specific.  Earthly
   maintains a [[https://github.com/earthly/buildkit][BuildKit fork]] but inherits the same core exec
   caching logic.  Any tool built on BuildKit has this limitation.

* Upstream status

  - [[https://github.com/dagger/dagger/issues/8955][Issue #8955]] — "Second to last step is not cached."  Root cause
    identified: =session:= prefix in LLB breaks BuildKit cache lookup.
  - [[https://github.com/dagger/dagger/issues/6421][Issue #6421]] — "=WithMountedDirectory= does not use content-based
    cache in =withExec=."  Related BuildKit limitation where
    content-based caching is explicitly disabled for non-readonly
    root mounts.
  - [[https://github.com/dagger/dagger/pull/11856][PR #11856]] — "Migrate all caching to dagql."  The planned fix:
    remove the BuildKit solver entirely and use dagql's
    content-addressed cache for everything.  Still WIP as of
    2026-04-15.

  When PR #11856 merges, the =include= filter approach already used
  in =src/dind.org= should work correctly — the exec cache will
  be content-based and identical filtered directories will share
  cache entries regardless of their operation history.

* Workarounds

  Until the upstream fix ships, the only way to get stable exec cache
  is to ensure the directory is constructed via the *same graph path*
  every time.  Two approaches:

** =DefaultPath= + =Ignore= with negation patterns

   The Dagger Python SDK supports =Ignore(["**", "!pattern", ...])=
   negation patterns (the SDK runtime itself uses this in
   =sdk/runtime/main.go=).  Combined with =DefaultPath(".")=, the
   engine constructs the directory from the context directory with
   the same filter every invocation.  When unrelated files change,
   the filtered directory keeps the same graph path because the
   filter inputs (non-ignored files) haven't changed.

   This works regardless of module layout — both =source: .dagger=
   and root-as-source configurations benefit (see the two
   sub-projects above).  Unrelated files at the root still trigger
   a module reload (~10 s) but the exec cache survives because the
   =Ignore= filter excludes them from the directory that =withExec=
   depends on.

   #+begin_src python :eval no-export :tangle no
   from typing import Annotated
   from dagger import DefaultPath, Ignore, function

   @function
   def my_function(
       self,
       src: Annotated[
           dagger.Directory,
           DefaultPath("."),
           Ignore(["**", "!src/", "!*.org", "!*.sh"]),
       ],
   ) -> dagger.Directory:
       ...
   #+end_src

   This is a per-function allowlist: each function declares exactly
   which files it needs, and unrelated file changes do not affect
   its cache.

** =dagger.json= =include= field

   The =include= field in =dagger.json= limits what Dagger considers
   the module source.  This is a module-level allowlist — simpler to
   maintain but shared across all functions:

   #+begin_src json :eval no-export :tangle no
   {
     "name": "my-module",
     "include": ["src/", "*.org", "*.sh", "dagger.json"]
   }
   #+end_src

* Impact on daggerlib

  The root =lib= module has no ="source"= field in =dagger.json= —
  its module source is the entire project root.  Five functions in
  [[file:../../src/dind.org][src/dind.org]] previously used =dag.current_module().source()= with
  per-function =include= filters, which busted the exec cache (~17 s
  for =dind_run_org=) whenever an unrelated file was added or removed
  at the project root.

  These functions now use =DefaultPath(".")= + =Ignore(["**",
  "!..."])= to declare per-function allowlists at the engine level:

  - =dind_run_tests=
  - =dind_tangle=
  - =dind_run_org=
  - =dind_init_examples=
  - =export_html=

  With this workaround, adding an unrelated file at the root still
  triggers a module reload (~7 s) but the exec cache survives —
  down from ~27 s to ~7 s.
