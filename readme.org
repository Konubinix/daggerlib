#+TITLE: Reusable container helpers for Dagger
#+LANGUAGE: en

daggerlib is a [[https://docs.dagger.io/features/programmable-pipelines][Dagger]] module that gives you pre-configured container
images for everyday tasks: setting timezones, creating non-root users,
managing Python virtualenvs, running Docker-in-Docker, and more.

Pick a base image, compose the helpers you need, and get a ready-to-use
container in a single =dagger call=.

*Note:* daggerlib is /not/ Dagger itself — it is a library of helpers
that builds on top of Dagger.

* What can I do with it?

  Most services I deploy run in France, so the first thing I need is a
  container with the correct timezone. [[file:src/alpine.org][Alpine]] is my default base image —
  it's small and fast to pull. When I need more packages or better
  compatibility with native libraries, I reach for [[file:src/debian.org][Debian]] instead.

  For production, I want the smallest possible attack surface.
  [[file:src/distroless.org][Distroless]] images have no shell and no package manager — just the
  runtime and the timezone file copied from Debian.

  Running as root inside containers is a bad habit. The [[file:src/user.org][user helpers]]
  let me create a non-root user with optional sudo and group memberships
  in a single call, on any base image.

  Python projects need virtualenvs. The Alpine and Debian targets can
  set one up automatically, but sometimes I need standalone tools:
  [[file:src/pip_tools.org][pip-tools]] for dependency management, or a [[file:src/flask_venv.org][pre-built Flask venv]] I can
  copy into a final image.

  Some CI jobs need to build Docker images. [[file:src/dind.org][Docker-in-Docker]] handles
  the cgroup setup, tmpfs mounts, and dockerd lifecycle so I don't have
  to think about it.

  For automated development tasks, [[file:src/ralph.org][Ralph]] runs the ralph-orchestrator
  inside a container with Claude Code, picking tasks from a todo list,
  implementing them, running tests, and committing — all without
  human intervention.

  All of this is verified by [[file:tests/testing.org][automated tests]] that exercise every target
  via =dagger call=.

* Dagger concepts

  If you're new to Dagger: instead of writing YAML pipelines, you write
  functions in Python (or Go, TypeScript, …) that describe what each
  container should look like. The Dagger SDK provides a fluent API to
  build and manipulate those containers. Here are the key primitives
  used in this project, with links to the [[https://docs.dagger.io/api/reference][API reference]]:

  - [[https://docs.dagger.io/api/reference#definition-Container][=dag.container()=]] — creates a new, empty container.
  - [[https://docs.dagger.io/api/reference#Container-from][=.from_(image)=]] — sets the base image (like =FROM= in a Dockerfile).
  - [[https://docs.dagger.io/api/reference#Container-withExec][=.with_exec([cmd])=]] — runs a command inside the container (like =RUN=).
  - [[https://docs.dagger.io/api/reference#Container-withEnvVariable][=.with_env_variable(name, value)=]] — sets an environment variable.
  - [[https://docs.dagger.io/api/reference#Container-withWorkdir][=.with_workdir(path)=]] — changes the working directory.
  - [[https://docs.dagger.io/api/reference#Container-withUser][=.with_user(username)=]] — switches the active user.
  - [[https://docs.dagger.io/api/reference#Container-withFile][=.with_file(path, file)=]] — copies a file into the container.
  - [[https://docs.dagger.io/api/reference#Container-file][=.file(path)=]] — extracts a single file from a container.
  - [[https://docs.dagger.io/api/reference#Container-directory][=.directory(path)=]] — extracts a directory from a container.
  - [[https://docs.dagger.io/api/reference#definition-Function][=@function=]] / [[https://docs.dagger.io/api/reference#definition-Module][=@object_type=]] — decorators that expose Python
    methods as callable Dagger functions (invokable via =dagger call=).
